# Двухэтапное ревью

> Методология проверки кода после реализации задачи/фичи.
> Применять после завершения плана или значимого блока работы (3+ файлов).

## Когда применять

- После реализации плана из `writing-plans.md`
- После рефакторинга, затрагивающего 3+ файлов
- Перед созданием PR
- По запросу пользователя

**Когда НЕ применять:**
- Однофайловый баг-фикс
- Изменение конфигов / стилей без логики

## Этап 1: Проверка спецификации

> «Сделано ли то, что было запрошено?»

### Чеклист

- [ ] **Цель достигнута** — фича работает как описано в плане/задаче
- [ ] **Все задачи выполнены** — ни одна задача из плана не пропущена
- [ ] **Сервисный слой** — Supabase вызовы только через `lib/*-service.ts`, паттерн `{ data, error }`
- [ ] **RLS учтён** — запросы работают с учётом Row Level Security (`.eq('user_id', ...)`)
- [ ] **Платформы** — работает на iOS, Android и Web (или задокументированы ограничения)
- [ ] **Нет лишнего** — не добавлены функции, которые не были запрошены

### Результат этапа 1

Если найдены расхождения со спецификацией — **исправить до перехода к этапу 2**.
Неправильный код нет смысла оптимизировать.

## Этап 2: Проверка качества кода

> «Сделано ли это правильно?»

### Дизайн-токены и стили
- [ ] Нет хардкод цветов — используются `Colors.*` из `constants/design-tokens.ts`
- [ ] Нет хардкод размеров — `borderRadius.*`, `spacing.*`, `FontNames.*`
- [ ] NativeWind классы корректны, кроссплатформенны

### Переводы
- [ ] Весь UI текст через `t('key', language)` из `constants/translations.ts`
- [ ] Нет хардкод строк в JSX
- [ ] Ключи добавлены для всех 4 языков (RU, UK, EN, DE)

### Типизация
- [ ] Типы определены в `types/` (не inline)
- [ ] `import type` для импорта только типов
- [ ] Нет `any`, `as any`, `@ts-ignore`

### Состояние (Zustand)
- [ ] Селективные подписки: `useStore(s => s.field)`, не `useStore()`
- [ ] Нет лишних ре-рендеров

### Безопасность
- [ ] Нет `service_role` ключей в клиентском коде
- [ ] Нет чувствительных данных в логах
- [ ] Только `EXPO_PUBLIC_*` переменные в клиенте
- [ ] Пользовательский ввод валидируется на границе сервисов

### Платформа
- [ ] `Platform.OS` проверяется где нужно (`Alert.alert` не работает на Web)
- [ ] Стили корректны на всех платформах

### Результат этапа 2

| Уровень | Описание | Действие |
|---------|----------|----------|
| **Блокер** | Безопасность, потеря данных, crash | Исправить обязательно |
| **Серьёзный** | Нарушение конвенций, утечка, неправильный тип | Исправить до завершения |
| **Замечание** | Стиль, именование, упрощение | Исправить по возможности |

## Формат вывода ревью

```
## Ревью: <Название задачи/фичи>

### Этап 1: Спецификация
✅ Цель достигнута
✅ Все задачи выполнены
⚠️ [Описание расхождения, если есть]

### Этап 2: Качество
✅ Дизайн-токены — ок
✅ Переводы — ок
⚠️ [Серьёзный] Описание проблемы
💡 [Замечание] Предложение по улучшению

### Итог
[Готово к коммиту / Требует исправлений (N блокеров, M серьёзных)]
```
