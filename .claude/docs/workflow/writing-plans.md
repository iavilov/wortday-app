# Декомпозиция задач

> Методология планирования перед реализацией.
> Читать при получении задач, требующих изменения 3+ файлов или создания новой фичи.

## Когда писать план

- Новая фича или экран (новая вкладка, геймификация, SRS)
- Интеграция с новым API / Supabase (новая таблица, RPC, Realtime)
- Рефакторинг, затрагивающий 3+ файлов
- Новый сервис в `lib/`

**Когда НЕ нужен план:**
- Баг-фикс в 1-2 файлах
- Изменение стилей / текстов
- Добавление перевода в `constants/translations.ts`

## Структура плана

```markdown
# <Название фичи>

## Цель
Одно предложение: что должно работать после реализации.

## Архитектура
2-3 предложения: какие слои затронуты (Supabase / lib / store / components / app),
какие паттерны используются, как данные текут.

## Задачи

### Задача 1: <Название> (~X мин)
**Файлы:** `src/path/to/file.ts` (новый|изменение)
**Шаги:**
1. Что сделать (конкретно)
2. ...

### Задача 2: ...

## Проверка
Как убедиться что всё работает (typecheck, визуальная проверка, конкретные сценарии).
```

## Правила декомпозиции

### Размер задачи: 5-15 минут
- Каждая задача = один логический шаг, который можно проверить изолированно
- Если задача больше 15 мин — разбить дальше
- Если задача меньше 5 мин — объединить с соседней

### Точные пути файлов
Каждая задача ОБЯЗАНА указывать:
- Какие файлы создаются (новый)
- Какие файлы изменяются (изменение)
- Какие файлы удаляются, если есть

### Порядок: зависимости → потребители
1. Сначала Supabase (миграции, RLS, RPC)
2. Типы (`types/`)
3. Сервисный слой (`lib/*-service.ts`)
4. Store (`store/*-store.ts`)
5. Компоненты (`components/`)
6. Экраны и навигация (`app/`)
7. Переводы (`constants/translations.ts`)

### Один план — одна фича
Не смешивать несколько несвязанных фич в одном плане.

## Выполнение плана

После создания плана — выполнять задачи **строго по порядку**:
1. Реализовать задачу
2. Убедиться что `npx tsc --noEmit` проходит (после каждых 2-3 задач)
3. Перейти к следующей задаче
4. После всех задач — полная проверка по `verification.md`

При обнаружении проблемы в плане — обновить план перед продолжением, а не импровизировать.
